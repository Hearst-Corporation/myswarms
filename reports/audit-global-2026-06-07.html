Code & Architecture6.5/10
P0 2P1 3P2 2

Le code TS est globalement propre et discipliné : zéro `any`/`@ts-ignore`/catch vide, validation Zod systématique à l'entrée des routes, mapping d'erreur engine cohérent (4xx propagé / 5xx→502), client HTTP centralisé avec retry/timeout, et frontière TS↔Python (`_internal.ts` + `swarms.ts`/`client.ts`) bien factorisée. Mais l'architecture d'auth a une faille concrète : la route `cockpit-chat` POST n'effectue AUCUN contrôle de propriété sur `chatId` (IDOR), et tout le modèle documenté repose sur un `src/middleware.ts` qui n'existe PAS dans `src/` (présent uniquement dans les worktrees). S'ajoutent une surface RCE-class (tools `write_file`/`run_command` exposés à un LLM via le chat) et un rate-limit in-memory inopérant en serverless qui ne couvre qu'un seul endpoint coûteux sur quatre.

P0 IDOR — POST /api/cockpit-chat n'autorise pas le chatId fourni par le client vérifié RÉEL → P0
src/app/api/cockpit-chat/route.ts
Quand le client envoie `body.chatId`, la route l'utilise directement (insert du message user ligne 55, exécution de l'agent puis insert de la réponse assistant ligne 90) SANS vérifier que ce chat appartient à `ownerId`. Toutes les écritures passent par `createAdminClient()` (service_role, bypass RLS total — admin.ts:8). Un utilisateur authentifié peut donc deviner/forcer un `chatId` d'un autre tenant, y injecter des messages ET déclencher l'agent (qui a accès à write_file, run_command, sql_query) dans le contexte d'un autre chat. À comparer avec GET/DELETE de cockpit-chats/[chatId]/route.ts qui filtrent correctement par `.eq("user_id", ownerId)` (lignes 21-22, 43) — l'incohérence confirme l'oubli côté POST.
🔧 Avant le 1er insert : si `body.chatId` est fourni, faire `admin.from('cockpit_chats').select('id').eq('id', chatId).eq('user_id', ownerId).single()` et retourner 404/403 si absent. Idéalement passer cette route à un client Supabase scellé par la session utilisateur (RLS) plutôt que service_role, comme prévu pour `sql_query`.

owner.ts:6-12 affirme « le middleware protège toutes les routes et garantit qu'un user est toujours présent » et « src/middleware.ts (updateSession — refresh session à chaque requête) » ; server.ts:32 répète « Le refresh de session est assuré par le middleware (updateSession) ». Or `find` confirme qu'il n'existe AUCUN `src/middleware.ts` dans l'arbre source (seulement des copies dans .claude/worktrees/*). 🔧 Recréer `src/middleware.ts` avec le pattern officiel @supabase/ssr `updateSession` (matcher excluant assets/_next), OU corriger les docstrings owner.ts/server.ts/superAdmin.ts pour refléter que la protection est 100% per-route et ajouter un test/lint garantissant que chaque route API appelle requireOwnerId/requireSuperAdmin.
P1 Surface RCE — tools write_file + run_command exposés à un LLM via le chat
src/lib/cockpit-agent/tools.ts
Le runtime agent (runtime.ts) branché sur POST /api/cockpit-chat exécute, à partir des tool_calls produits par Kimi, des écritures de fichiers arbitraires dans le projet (writeFileTool, l.316) et des commandes shell via `spawn` (runCommandTool, l.332-385). La protection repose uniquement sur une denylist de tokens (scope.ts:65 — `rm`, `|`, `&&`, `$(`, …) et une allowlist de binaires (npm/npx/node/git…). C'est contournable (npm/npx peut exécuter des scripts arbitraires : `npx <package>` ; `node script.js` ; un `package.json` réécrit par write_file puis `npm run`). Combiné au P0 IDOR ci-dessus, un attaquant peut piloter ces tools dans un contexte arbitraire. La séparation client/serveur est respectée (server-only), mais le couplage chat→exécution-système est une dette d'architecture majeure pour une app multi-tenant.
🔧 Désactiver write_file/run_command en production (gate NODE_ENV comme system/start/route.ts:32), ou les réserver au super-admin via un flag, ou sandboxer (conteneur jetable, FS read-only). Au minimum : passer d'une denylist à une allowlist stricte d'argv et interdire npx/node/`npm run` arbitraires.

P1 Inserts cockpit_messages non gardés : message user persisté même si chat invalide / insert échoué

L'insert du message user (l.55-59) est `await` mais son `error` n'est jamais vérifié — si l'insert échoue (FK chat_id invalide, contrainte), la requête continue, lance l'agent (coût LLM) et renvoie 200 avec une conversation incohérente. La gestion d'erreur du reste du fichier est correcte (try/catch JSON l.28, runAgent l.97) — c'est l'absence de check sur le résultat Supabase qui manque.

P2 authedFetch : assertion non-null lastRes! et AbortError non distingué

La boucle de retry retourne `lastRes!` (l.114) avec assertion non-null. La logique l.97-100 garantit en pratique un retour avant d'atteindre l.114 (le dernier attempt retourne `res` directement l.100), donc `lastRes` ne devrait jamais être undefined — mais l'assertion `!` masque le raisonnement et casserait silencieusement si la condition l.100 évoluait. P1 Isolation multi-tenant de l'engine fondée sur un owner_id non vérifié (confiance totale dans le BFF)
services/crewai-engine/src/routes/swarms.py:45
Depuis la migration 0026 (drop FK owner_id -> auth.users), l'engine n'a plus aucun lien avec une auth réelle. _require_owner_id() (swarms.py:45) valide seulement que owner_id est un UUID bien formé — pas qu'il appartienne à l'appelant. L'auth engine = un unique bearer partagé (CREWAI_ENGINE_AUTH_TOKEN) commun à tous les tenants/clients du BFF. Donc quiconque possède ce bearer (le BFF, mais aussi tout service qui le réutilise, ou en cas de fuite) peut passer N'IMPORTE QUEL owner_id en query-param et lire/écrire/kicker les swarms et runs de n'importe quel tenant. 🔧 Côté engine, dériver/valider l'owner_id depuis un JWT vérifiable plutôt que de le faire confiance en clair (ex : propager le JWT Supabase de l'utilisateur et le valider, ou signer un token par-tenant côté BFF). A minima : rendre ownerId obligatoire (non-optionnel) dans le client TS swarms.ts et documenter le bearer engine comme secret de très haute valeur (rotation, scope réseau restreint à Vercel).

P2 exec_readonly_sql : filtrage DML par regex contournable (defense-in-depth faible)
supabase/migrations/0017_exec_readonly_sql.sql:13
exec_readonly_sql bloque les mots-clés DML via une regex `\m(insert|update|...)\M` puis `set transaction read only`. La regex est fragile (commentaires SQL, casing exotique, CTE writable masquées) ; heureusement `set transaction read only` est le vrai garde-fou côté Postgres et empêche toute écriture. Le risque résiduel réel n'est donc pas l'écriture mais la lecture exhaustive (couplée au bypass RLS du P0 sql_query). En contexte user-scoped correctement appliqué, l'impact est limité.
🔧 Conserver `set transaction read only` comme garde principal. Optionnel : remplacer la regex par une dépendance stricte sur read-only transaction + role en lecture seule dédié, et documenter que la regex n'est qu'un message d'erreur précoce, pas une frontière de sécurité.
